Before exploiting, I enumerated the application further for keyspaces, but I didn't find anything.

Let's summarise our attack vector potentials. The target machine is running SSH application on port 22, but I do not have any potential usernames & passwords. The target machine is also running Redis 4.0.14 which is vulnerable to the infamous Redis Rogue Server exploit. And this attack vector looks the most promising. Time to try it out.

Executing the binary using the command sudo -u root /usr/local/bin/redis-status informs me to enter the authorisation key. I do not have any sort of password so far. But since this is a binary, maybe I can run strings on it to obtain what the binary wants.


Running the command strings /usr/local/bin/redis-status returns the result shown in the image above. I found the authorisation key ClimbingParrotKickingDonkey321 which the binary is looking for.
